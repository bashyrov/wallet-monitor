# Kickoff-промпт для агента (Claude Code / Cursor)

Ты работаешь в репозитории Avalant (Go-датаплейн `go-fetcher` + Python FastAPI backend).
Задача: максимально ускорить фетчинг стаканов — поднять частоту обновления на клиенте
с текущих ~5/сек до 20–30+/сек, как у конкурентов, выжав максимум скорости из КАЖДОЙ биржи.

ПЕРВЫМ ДЕЛОМ прочитай два файла в корне и следуй им как источнику истины:
- ORDERBOOK_OPTIMIZATION.md — полный анализ + per-provider целевые каналы (Раздел 3) + порядок.
- STATUS.md — рабочий журнал. Ты его ведёшь и обновляешь после КАЖДОГО изменения.
  В STATUS.md есть заполненный ПРИМЕР (Раздел C) — повторяй его формат.

ГЛАВНОЕ ПОНИМАНИЕ:
Частота обновлений на клиенте = МИНИМУМ из тактов цепочки (правило слабого звена):
канал биржи → cache → Redis throttle (50ms=20/с) → flushLoop (200ms=5/с) → браузер.
Сейчас связывает flushLoop = 5/сек. Это min()-цепочка: поднимать надо ВСЕ звенья,
иначе потолок просто переезжает на следующее. Меняешь по одному и смотришь, куда он уехал.

ПРАВИЛА РАБОТЫ:
1. Измеряй, не угадывай. Метрика — updates/sec на клиенте (/ws/book, фреймы за 10s ÷ 10).
   Меряй в 3 точках (клиент / источник в ws/runner.go / выход broadcaster) — см. Раздел 7.2.
2. СНАЧАЛА сними baseline (binance:BTC, okx:ETH, одна низколиквидная) и замерь конкурента
   тем же методом. Запиши в STATUS.md разделы A и B. НЕ трогай код до baseline.
3. Меняй СТРОГО по одному параметру. После каждого — замер тех же пар, записать before/after
   В ЦИФРАХ и куда переехал потолок. Без after-числа задача НЕ done.
4. Каждое изменение — за env/фиче-флагом (AVALANT_BOOK_FLUSH_INTERVAL,
   AVALANT_REDIS_WRITE_THROTTLE, <VENUE>_USE_BBO), чтобы откатывать и A/B-тестить по бирже.
5. После смены канала ОБЯЗАТЕЛЬНО проверь корректность (Раздел 5): best bid < best ask,
   1 уровень BBO корректно ложится в []Level, reconnect сбрасывает state. Кривой стакан
   хуже медленного. Регрессию фиксируй в STATUS.md (колонка bid<ask + раздел H).
6. Порядок: Фаза 1 (фронт-проверка → flushLoop → throttle → reconcile → mtime-skip),
   потом Фаза 2 (каналы по одной бирже по таблице 3.3 и E), потом Фаза 3 (архитектура).

PER-PROVIDER ЦЕЛИ (точные каналы — в Разделе 3 дока, кратко):

не менять вслепую.
