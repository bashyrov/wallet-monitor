## Автономный цикл (повторяй для каждой задачи)

```
1. Прочитай STATUS.md → возьми следующую задачу по приоритету.
2. Если задаче нужен канал/формат подписки, которого ты не знаешь ТОЧНО:
   → fetch официальную WS-документацию биржи, подтверди точный topic/subscribe/payload.
   → НЕ внедряй неподтверждённый формат. Не уверен после доков → mark blocked, в STATUS,
     переходи к следующей задаче.
3. Внедри ОДНО изменение, спрятав за env/фиче-флагом (<VENUE>_USE_BBO и т.п.).
4. Собери, прогони тесты адаптера. Замерь в 3 точках (клиент / источник / broadcaster), §7.2.
5. Регрессия (§5): best bid < best ask; 1 уровень BBO корректно в []Level; reconnect
   сбрасывает state; seq gap → resync. Если сломалось — откати флагом, запиши в H, чини.
6. Запиши before/after В ЦИФРАХ в STATUS.md + куда переехал потолок. Без числа = НЕ done.
7. git commit с осмысленным сообщением (одно изменение = один коммит).
8. → следующая задача. НЕ спрашивай разрешения между задачами.
```

## Правила самостоятельного фикса поломок

- Сломанная/нулевая биржа: локализуй (коннект → subscribe/REST → recv → Parse) счётчиками и логами,
  потом сверь формат с ОФИЦИАЛЬНОЙ докой биржи, почини, протестируй, зафиксируй.
- Чинишь по доке, а не по догадке. Если дока не разрешает однозначно → blocked + заметка, дальше.
- После любого фикса — тот же замер и регрессия-чек, что и для обычной задачи.

## Жёсткие правила (не нарушать)

- Одно изменение за раз. Никогда не меняй несколько параметров/бирж в одном шаге.
- done только с after-замером в STATUS.md. «Сделал» без числа = не сделано.
- Не выдумывай каналы. Любой новый канал — только после подтверждения в доках.
- Каждое изменение за флагом (откат без редеплоя, A/B по бирже).
- Регрессия важнее скорости: кривой стакан хуже медленного.
- Сервер/железо/локация — вне скоупа, не трогай.
- min()-цепочка: после каждого шага явно фиксируй, какое звено теперь связывает.

## Решения принимаешь САМ. К человеку не возвращаешься.

Не останавливайся и не жди меня ни на чём. В каждой ситуации действуй сам по правилам ниже,
ОБЯЗАТЕЛЬНО фиксируя решение и обоснование в STATUS.md, и продолжай очередь.

- **Неожиданная деградация, решение не найдено после диагностики:** ОТКАТИ флагом к рабочему
  before-состоянию (метрика не должна стать хуже, чем была). Статус `reverted`, в STATUS.md
  раздел H запиши: что пробовал, какая деградация, что вернул. Продолжай со следующей задачи.
  Лучше сохранить рабочее «как было», чем застрять.

- **Дока биржи противоречива/недоступна:** прими best-effort решение сам — выбери наиболее
  вероятный по другим источникам/SDK канал, пометь в STATUS «не подтверждено офиц. докой»,
  внедри за флагом и протестируй. Взлетело + регрессия чистая → оставь, помечай риск.
  Не взлетело → откати, статус `blocked` + причина, дальше. Не зацикливайся.

- **Архитектурное решение (KuCoin split, gRPC и т.п.):** выбери САМ наименее рискованный
  вариант, который можно сделать за выключаемым флагом и откатить. Кратко запиши в STATUS.md
  рассмотренные варианты + почему выбрал этот. Сделай, замерь, проверь регрессию. Хуже стало
  или не заработало → откат, статус `reverted`/`blocked` + причина, дальше.

- **Закончил всю очередь:** НЕ жди меня. Запиши в STATUS.md сводный финальный отчёт: метрика по
  каждой бирже vs цель 20–30/с, что осталось `blocked`/`reverted` и почему, какие риски
  «не подтверждено докой». На этом останавливайся — отчёт это финал, не вопрос.

Главный принцип: всегда оставляй систему в рабочем состоянии не хуже предыдущего, документируй
решение, и иди дальше. Откат + честный статус «не удалось» — это валидный исход, а не повод
останавливать работу или звать человека.
